A reasonable question, in 2026: if AI assistants can stub out a Terraform module, scaffold a docs site, or draft a slide deck on demand, does a tool like Dinghy still earn its place in the stack?

Short answer: yes, more so than before. Long answer below — including the awkward statistic that **80% of Dinghy's own codebase is generated**.

<!-- truncate -->

## Is Dinghy still relevant?

People still write code — they just write it with AI help. That has not changed the fundamental problem: codebases need *structure* to be maintainable, and structure is exactly what abstractions provide.

Dinghy's pitch in this world is not "we write the code for you". It is:

- **Generation** — Dinghy gives the AI a stable target. TSX components, YAML DSLs, opinionated defaults. The AI fills in the parameters; the structure is already there.
- **Maintenance** — when the AI gets it wrong, the diff is bounded. You are reviewing 8 lines of TSX, not 248 lines of Terraform JSON.
- **Software development lifecycle** — Dinghy outputs are *real artifacts*: Terraform JSON, draw.io files, Docusaurus sites. They go through normal CI / review / deploy paths, not vibes.

In other words: Dinghy is the layer that makes AI-generated infrastructure *trustworthy* enough to ship.

## 800K lines of code (as of Feb 2026)

Dinghy's own codebase is a useful case study. The breakdown:

- 🛠️  **15% hand-crafted** — the engine, the renderer, the CLI wiring. The hard parts.
- ⚙️  **80% generated by CodeGen** — bindings, providers, AWS resource icons. The repetitive parts.
- 🤖  **5% with AI assistant** — feature scaffolding, edge-case fixes. Every line reviewed.

The 80% number is the interesting one. It is not AI; it is **code generation** in the boring, reliable sense — schemas in, code out. The 5% AI slice is layered on top, and it is small on purpose: every line is reviewed before it ships.

The result is a codebase that scales like it has 800K lines of effort but maintains like it has 120K lines of hand-written code, because the rest is regenerated whenever the source schemas change.

## Dinghy + AI assistants

Dinghy ships **skills** for the major AI coding assistants — Claude Code, Copilot, Gemini. A skill is a focused capability, accessed by name:

- `/dinghy-iac` — scaffold a Dinghy IaC project.
- `/dinghy-diagram` — author an architecture diagram.
- `/dinghy-site` — spin up a Docusaurus site.
- `/dinghy-slide` — generate a slide deck.

A real example, the one that makes this concrete:

```text
/dinghy-slide
create a slide deck about Dinghy,
  use https://dinghy.dev/guides/get-started
  and its sub pages as source material
```

The skill knows the Dinghy slide format. The assistant knows how to read source material. The output is a YAML/Markdown deck you can preview locally and refine — not a generic blob of text the AI invented from scratch.

To install the skills:

```bash
dinghy skill
```

That command installs the skills for the popular AI assistants you have configured.

## The pattern

The Dinghy + AI partnership is the same shape as Dinghy + a human:

- The toolchain provides the **abstractions**.
- The author provides the **intent**.
- The output is something **structured** that fits into the rest of your software lifecycle.

Whether the author is a human, an AI, or a human-with-AI, the structure is what makes the result reviewable and shippable. That does not get less valuable as AI gets better — it gets *more* valuable, because the volume of generated code goes up and the cost of unstructured generation goes up with it.
