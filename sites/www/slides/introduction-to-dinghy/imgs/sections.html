Under the Hood

How Dinghy works

Press ↓ for details

Let's look at the layers underneath the friendly CLI.

Three-layer architecture

1
CLI (Docker)

The stable, versioned entry-point you install once. Drives everything else.

2
Engine (Deno runtime)

Zero language configuration. Performs the critical render operations — fast, reliable, predictable.

3
React templating

Your data model lives in TSX components. The engine renders them into the target format.

Key insight: the CLI is thin — it just calls the versioned Engine. The Engine does the heavy lifting with Deno (no Node, no npm install needed on the host).
Step 1

Quick Install

One command — that's it.

$ curl -fsSL https://get.dinghy.dev/install.sh | sh

🐳 Requires Docker on your system.
Having trouble? Check the Installation → Prerequisites section in the docs.

The install script pulls the Dinghy CLI wrapper onto your system. Docker does the actual heavy lifting so there are no language runtime conflicts.
Step 2

Write your first diagram

Create my-simple-diagram.tsx

import { Shape } from '@dinghy/base-components'
 
export default () => (
  <Shape _title='Simple diagram as code'>
    <Shape _dependsOn='B'>A</Shape>
    <Shape>B</Shape>
  </Shape>
)

  <Shape _title='Simple diagram as code'>
    <Shape _dependsOn='B'>A</Shape>
    <Shape>B</Shape>
  </Shape>
)
Three key things: import Shape from the base component library, export a default function component, nest Shapes to describe relationships. _dependsOn declares a dependency edge.
Step 3

Render

From the same directory, run:

$ dinghy render
One command. Dinghy picks up all TSX files in the directory and renders each one.
Step 3

Render

From the same directory, run:

$ dinghy render
✓ Output — files written to the output/ folder:
  • my-simple-diagram.png — the rendered image
  • my-simple-diagram.drawio — editable draw.io source
Dinghy writes both a PNG (for embedding in docs/tickets) and the raw draw.io XML you can open and edit manually if needed.
Result

What you get

The simple TSX above produces a draw.io diagram showing Node A depending on Node B:

  • Each <Shape> becomes a box
  • _dependsOn draws an arrow
  • _title labels the diagram
The diagram maps 1-to-1 with the component structure. Real-world diagrams will use more expressive custom components — but the mental model stays the same.
